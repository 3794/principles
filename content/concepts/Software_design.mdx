# Software Design

소프트웨어 디자인이란 시스템을 구현하기 전에 개념적으로 구상하는 process이면서, 그 process의 결과물을 의미한다.

#### Method, Approach, and Styles

- Domain Driven Design (DDD)
  - 도메인 주도 설계: 복잡한 소프트웨어를 실제 비즈니스 도메인(업무 영역)의 개념과 규칙에 맞춰 설계하는 방법입니다. 도메인 모델을 중심에 두고, 코드와 비즈니스의 일치성을 높입니다.
- Test Driven Development (TDD)
  - 테스트 주도 개발: 실제 코드를 작성하기 전에 테스트 코드를 먼저 작성하고, 이 테스트를 통과하는 코드를 개발하는 방식입니다. 코드 품질과 유지보수성을 높입니다.
- Behavior Driven Development (BDD)
  - 행위 주도 개발: 요구사항을 사용자(비즈니스) 관점에서 자연어에 가깝게 정의하고, 그 행위(Behavior)를 테스트로 검증하는 개발 방식입니다. 개발자와 비즈니스 담당자 간의 커뮤니케이션을 강화합니다.
- Event Driven Design/Development (EDD)
  - 이벤트 주도 설계/개발: 시스템 내의 주요 상태 변화나 이벤트를 중심으로 설계합니다. 이벤트가 발생하면 그에 따라 시스템이 반응하도록 구조화합니다. 대규모 분산 시스템, 실시간 처리에 적합합니다.
- Data Driven Design
  - 데이터 주도 설계: 데이터의 구조와 흐름을 중심으로 시스템을 설계합니다. 데이터베이스 중심 애플리케이션, 데이터 분석 시스템 등에 주로 사용됩니다.
- Contract Driven Design
  - 계약 주도 설계: 서비스 간의 인터페이스(계약, 예: API 스펙)를 먼저 정의하고, 이 계약에 맞춰 개발을 진행하는 방식입니다. 마이크로서비스, API 중심 시스템에서 활용됩니다.
- Model Driven Design (MDD)
  - 모델 주도 설계: UML 등 모델링 언어로 시스템 구조를 먼저 정의하고, 모델을 바탕으로 실제 코드를 자동 생성하거나 개발을 진행합니다.
- Dependency Driven Design
  - 의존성 주도 설계: 시스템 내 모듈 간의 의존성 관리와 설계를 중시하는 접근법입니다. 느슨한 결합, 높은 응집도를 목표로 합니다.
- [Object-Oriented Design (OOD)](/concepts/Object-Oriented_Design)
  - 객체지향 설계: 소프트웨어 시스템을 객체(Object)라는 단위로 나누어, 이 객체들이 서로 협력하며 전체 시스템을 구성하도록 설계하는 방법론입니다.
