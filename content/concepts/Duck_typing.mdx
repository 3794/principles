# Duck Typing

**덕 타이핑(Duck Typing)** 은 동적 타입 언어에서 객체의 타입을 확인하는 대신, **필요한 메서드나 속성을 가지고 있는지**를 확인하는 방식이다.

이름은 다음 문장에서 비롯되었다:

> "만약 그것이 오리처럼 보이고, 오리처럼 울고, 오리처럼 헤엄친다면, 그것은 오리다."

즉, 객체의 **실제 타입(클래스)** 은 중요하지 않고, **필요한 행동을 할 수 있는가**가 중요하다.

## Core Concept

덕 타이핑은 [Substitutability](/concepts/Substitutability)를 **런타임에 동적으로 검증**하는 방식이다.

```python
# 어떤 타입이든 상관없음
# "quack()" 메서드가 있으면 오리로 취급
def make_it_quack(bird):
    bird.quack()  # 메서드가 있으면 실행, 없으면 에러
```

**특징:**

- 컴파일 타임 타입 체크 없음
- 메서드 존재 여부로 판단
- 명시적 상속/인터페이스 구현 불필요
- 런타임 에러 가능성

## Simple Example

### 정적 타입 언어 vs 동적 타입 언어

```typescript
// TypeScript (정적 타입)
interface Drawable {
  draw(): void;
}

class Circle implements Drawable {
  draw() {
    console.log("원을 그립니다");
  }
}

class Square implements Drawable {
  draw() {
    console.log("사각형을 그립니다");
  }
}

function render(obj: Drawable) {
  obj.draw(); // 타입 체크: Drawable 구현 확인
}
```

```python
# Python (덕 타입)
class Circle:
  def draw(self):
    print("원을 그립니다")

class Square:
  def draw(self):
    print("사각형을 그립니다")

class Triangle:
  def draw(self):
    print("삼각형을 그립니다")

def render(obj):
  obj.draw()  # 타입 체크 없음, 메서드만 있으면 OK

# 상속 관계 없음!
render(Circle())      # OK
render(Square())      # OK
render(Triangle())    # OK
```

## 런타임 에러

덕 타이핑의 위험성: 필요한 메서드가 없으면 **런타임에 에러 발생**

```python
class Rock:
  pass  # fly() 메서드가 없음!

try:
  start_flight(Rock())  # AttributeError 발생!
except AttributeError as e:
  print(f"에러: {e}")  # 에러: 'Rock' object has no attribute 'fly'
```

## 덕 타이핑의 장점

- **유연성** - 상속 계층이 필요 없음
- **간단함** - 인터페이스 명시 불필요
- **빠른 개발** - 타입 선언 줄임
- **다형성** - 자동으로 다형성 구현

## 덕 타이핑의 단점

- **타입 불안전** - 컴파일 타임 검사 불가
- **런타임 에러** - 실행 중에야 문제 발견
- **문서화 필요** - 필요한 메서드를 명시해야 함
- **IDE 지원 부족** - 자동 완성, 리팩토링 어려움

## 타입 힌트로 개선

최근 Python, JavaScript는 **타입 힌트(Type Hints)** 로 보완한다.

```python
# Python 3.5+: 타입 힌트
from typing import Protocol

class Flyable(Protocol):
  def fly(self) -> str: ...

def start_flight(flying_object: Flyable) -> None:
  print(flying_object.fly())

# 이제 타입 체커(mypy)가 컴파일 타임에 검사 가능
```

## Substitutability와의 관계

덕 타이핑은 **[Substitutability](/concepts/Substitutability)의 동적 구현**이다.

| 타입 시스템     | 대체 가능성 검증 | 메커니즘               |
| --------------- | ---------------- | ---------------------- |
| 정적 타입 (OOP) | 컴파일 타임      | 명시적 상속/인터페이스 |
| 구조적 타입     | 컴파일 타임      | 구조 일치              |
| 덕 타입         | 런타임           | 메서드 존재            |

**모두 같은 목표: 대체 가능성을 정의하고 검증하기**

## 덕 타이핑이 잘 맞는 상황

- 프로토타이핑 또는 초기 개발
- 작은 규모의 스크립트
- 팀이 동적 타입 언어에 능숙함
- 타입 안전성보다 유연성이 중요함

## 덕 타이핑이 맞지 않는 상황

- 큰 규모의 프로젝트
- 팀 규모가 크거나 변함
- 높은 안정성이 필요함
- 오래 유지보수해야 할 코드
