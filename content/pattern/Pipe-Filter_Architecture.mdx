# Pipe-Filter Architecture

파이프-필터 아키텍처는 데이터가 여러 단계를 거치면서 순차적으로 처리되는 구조다. 유닉스의 파이프라인 개념과 유사하다.

```bash
cat file.txt | grep "error" | sort | uniq -c
```

이처럼 데이터가 한 방향으로 흐르면서 각 단계에서 변환되는 구조다.

## 아키텍처 구조

```
┌────────┐   ┌────────┐   ┌────────┐   ┌────────┐
│ input  │ → │filter A│ → │filter B│ → │ output │
│(Source)│   │        │   │        │   │ (Sink) │
└────────┘   └────────┘   └────────┘   └────────┘
```

## 핵심 구성요소

### 파이프 (Pipe)

데이터가 흐르는 통로다.

- 단방향으로만 흐름
- 각 필터를 연결하는 역할
- Point-to-Point 방식 (필터 A의 출력 = 필터 B의 입력)

### 필터 (Filter)

데이터를 처리하는 단계다. 네 가지 종류가 있다.

| 종류        | 역할                  | 예시                   |
| ----------- | --------------------- | ---------------------- |
| Producer    | 데이터 생성/입력      | 파일 읽기, API 호출    |
| Transformer | 데이터 변환           | JSON→XML, 암호화       |
| Tester      | 조건 검사, 필터링     | 유효성 검사, 중복 제거 |
| Consumer    | 최종 데이터 저장/출력 | DB 저장, 화면 출력     |

## 실제 사례

### ETL 파이프라인

```
[데이터 추출] → [정제] → [변환] → [적재]
 Extract       Clean   Transform   Load
```

로그 분석 시스템 예시:

```
원본 로그 → 불필요한 로그 제거 → 포맷 변환 → 통계 계산 → DB 저장
```

### 컴파일러

```
소스코드 → 렉싱 → 파싱 → 의미분석 → 최적화 → 코드생성 → 실행파일
```

### 데이터 처리 시스템

```
센서 데이터 수집 → 노이즈 제거 → 단위 변환 → 이상값 탐지 → 저장
```

### CI/CD 파이프라인

```
소스코드 커밋 → 빌드 → 테스트 → 배포
```

## 핵심 장점

파이프라인이 강력한 이유는 각 필터가 완전히 독립적이기 때문이다.

### 재사용성

암호화 필터를 여러 파이프라인에서 재사용 가능하다.

### 교체 용이성

필터 B를 다른 필터로 교체해도 나머지 필터에 영향이 없다.

### 병렬 처리

독립적인 필터들을 동시에 실행할 수 있다.

### 테스트 용이성

각 필터를 독립적으로 테스트할 수 있다. 입력과 출력만 검증하면 된다.

## Layered Architecture와의 비교

| 구분          | Layered            | Pipe-Filter      |
| ------------- | ------------------ | ---------------- |
| 데이터 흐름   | 양방향 가능        | 단방향           |
| 컴포넌트 관계 | 레이어 간 의존     | 필터 간 독립     |
| 주요 용도     | 일반 웹 앱         | 데이터 처리      |
| 상태 관리     | 레이어가 상태 유지 | 필터는 무상태    |
| 변경 용이성   | 레이어 전체 영향   | 해당 필터만 영향 |

## 무상태 필터 원칙

파이프라인 아키텍처에서 필터는 무상태(Stateless)여야 한다. 이것이 독립성과 재사용성의 핵심이다.

### 올바른 필터

```
입력을 받아 → 처리하고 → 출력을 내보낸다
```

이전 데이터를 기억하지 않는다.

### 잘못된 필터

```
이전에 처리한 데이터를 기억하고 그것에 따라 다르게 처리한다
```

이는 필터 간 의존성을 만들어 파이프라인의 장점을 사라지게 한다.

## 적용 시기

### ✅ 적합한 상황

- 데이터를 순차적으로 변환해야 할 때
- ETL 작업
- 배치 처리 시스템
- 컴파일러, 언어 처리기
- 이벤트 로그 처리
- AI/ML 데이터 전처리 파이프라인
- 이미지/비디오 처리

### ❌ 부적합한 상황

- 실시간 응답이 필요한 시스템
- 데이터 흐름이 복잡하고 양방향인 시스템
- 높은 가용성이 필요한 서비스
- 트래픽이 급변하는 시스템

## 최근 사례

파이프라인 아키텍처는 현대 소프트웨어에서 매우 활발하게 사용된다.

- **Apache Kafka** — 이벤트 스트리밍 파이프라인
- **Apache Spark** — 대용량 데이터 처리 파이프라인
- **GitHub Actions** — CI/CD 파이프라인
- **AWS Step Functions** — 클라우드 워크플로우
- **ML 파이프라인** — 데이터 수집 → 전처리 → 학습 → 배포

특히 데이터 엔지니어링과 ML 분야에서는 사실상 표준이다.
