# Data Flow Model

데이터 흐름 모델은 시스템에서 데이터가 생산자(Producer)에서 소비자(Consumer)로 이동하는 방식을 정의한다. Push와 Pull이라는 두 가지 근본적인 패턴이 있다.

## Push Model (발행-구독)

생산자가 능동적으로 데이터를 소비자에게 전달한다.

### 특징

- 생산자가 데이터 발생 시 즉시 전송
- 소비자는 수신을 대기
- 생산자가 흐름을 제어

### 예시

```
Producer → Event/Message → Consumer
```

이벤트 기반 아키텍처, 메시지 큐, 발행-구독 패턴에서 주로 사용된다.

### 장점

- 실시간 처리 가능
- 이벤트 발생 즉시 반응

### 단점

- 소비자가 처리 속도를 따라가지 못하면 메시지 손실 가능
- 생산자는 소비자의 상태를 알 수 없음

### 적합한 상황

- 실시간 알림이 필요할 때
- 다수의 소비자가 같은 데이터를 처리해야 할 때
- 이벤트 기반 시스템

## Pull Model (요청-응답)

소비자가 필요할 때 능동적으로 생산자에게 데이터를 요청한다.

### 특징

- 소비자가 데이터 필요 시 요청
- 생산자가 응답
- 소비자가 흐름을 제어

### 예시

```
Consumer → Request → Producer → Response → Consumer
```

REST API, 데이터베이스 쿼리, 폴링에서 주로 사용된다.

### 장점

- 소비자가 처리 속도를 제어 가능
- 필요한 데이터만 요청 가능
- Backpressure 자연스럽게 처리됨

### 단점

- 지연 발생 가능 (실시간성 낮음)
- 소비자가 주기적으로 요청해야 함 (폴링의 경우 비효율)

### 적합한 상황

- 클라이언트가 언제 데이터가 필요한지 알 때
- API 기반 통신
- 데이터 일관성이 중요할 때

## 비교

| 특성      | Push                         | Pull       |
| --------- | ---------------------------- | ---------- |
| 제어권    | 생산자                       | 소비자     |
| 실시간성  | 높음                         | 낮음       |
| 복잡도    | 높음 (처리 못한 데이터 관리) | 낮음       |
| 역압 처리 | 어려움                       | 쉬움       |
| 지연      | 적음                         | 클 수 있음 |

## Hybrid: Push-Pull

실제 시스템에서는 두 모델을 결합하기도 한다.

### React 아키텍처와 Push/Pull

React는 겉으로 보면 **Push** 모델 처럼 보인다.

```typescript
// 부모 컴포넌트가 상태를 변경하면 자식 컴포넌트에 push 되는 것처럼 보인다.
function Parent() {
  const [count, setCount] = useState(0);

  return <Child count={count} />;
}

function Child({ count }) {
  return <p>{count}</p>;
}
```

그러나 내부적으로는 pull 모델로 구성되어 있다.

```
// 상태가 바뀌면, UI를 다시 계산한다.
UI = f(state)
```

이 구조 때문에 실제 렌더링 단계에서는

- 컴포넌트가 현재 state를 pull로 읽고
- 컴포넌트를 다시 실행해서 새로운 UI를 만든다.

그래서 정리하면
| 관점 | 설명 |
|---|----|
|트리거| push |
|렌더링 | pull |
