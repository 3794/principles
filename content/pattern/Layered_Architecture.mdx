# Layered Architecture

레이어드 아키텍처는 가장 흔하고 직관적인 아키텍처 스타일이다.

## 계층 구조

```
┌────────────────────┐
│ Presentation Layer │ ← 사용자 인터페이스, API
├────────────────────┤
│ Business Layer     │ ← 비즈니스 로직
├────────────────────┤
│ Persistence Layer  │ ← DB 쿼리, ORM
├────────────────────┤
│ Database Layer     │ ← 실제 데이터베이스
└────────────────────┘
```

각 레이어는 자신의 바로 아래 레이어하고만 통신하는 것이 원칙이다.

## 각 레이어의 역할

### Presentation Layer

- HTTP 요청/응답 처리
- 입력값 유효성 검사
- UI 렌더링
- 비즈니스 로직은 절대 없어야 함

### Business Layer

- 핵심 비즈니스 로직
- 규칙 적용 (할인 계산, 재고 확인 등)
- 여러 Persistence를 조합하는 로직
- 가장 중요한 레이어

### Persistence Layer

- SQL 쿼리 작성
- ORM 매핑
- DB 연결 관리
- 데이터 접근만 담당, 비즈니스 로직 없어야 함

### Database Layer

- 실제 데이터 저장소
- Oracle, MySQL, PostgreSQL 등

## 레이어 격리 (Layer Isolation)

레이어드 아키텍처의 핵심 원칙은 "각 레이어는 자신의 역할만 알고, 다른 레이어의 내부를 알면 안 된다"는 것이다.

### 격리된 레이어 (Closed Layer)

```
Presentation → Business → Persistence → Database
```

반드시 순서대로만 통신한다. Presentation이 Persistence를 직접 호출하면 안 된다.

### 개방된 레이어 (Open Layer)

가끔 레이어를 건너뛰는 게 허용될 때가 있다. 예를 들어 공통 유틸리티 레이어는 어디서든 접근 가능할 수 있다. 레이어를 열어둘 때는 반드시 문서화해야 한다. 그렇지 않으면 왜 건너뛰는지 아무도 모른다.

## 주요 안티패턴

### 싱크홀 안티패턴 (Sinkhole Anti-pattern)

요청이 들어왔을 때 Business 레이어에서 아무 처리도 하지 않고 그냥 Persistence로 넘기기만 하는 상황이다.

**예시**:

```
사용자가 이름을 조회한다
Presentation: "Business야, 이름 조회해줘"
Business: "알겠어" (아무 처리 없이)
Persistence: DB에서 조회해서 반환
Business: 그대로 반환
Presentation: 그대로 반환
```

Business 레이어가 아무 역할도 하지 않으면 레이어가 존재할 이유가 없다. 그냥 복잡성만 추가하는 꼴이다.

**적정 비율**:

- 싱크홀 요청이 전체의 20% 이하 → 허용 가능
- 싱크홀 요청이 전체의 80% 이상 → 레이어드가 맞지 않는 시스템

### 아키텍처 오염 (Architecture Pollution)

비즈니스 로직이 잘못된 레이어에 들어가는 문제다.

```
❌ 잘못된 예:
Presentation Layer에서 할인 계산을 함
→ UI가 바뀌면 비즈니스 로직도 같이 바뀌어야 함

❌ 또 다른 잘못된 예:
Persistence Layer에서 비즈니스 규칙을 적용함
→ DB 기술이 바뀌면 비즈니스 로직도 바뀌어야 함
```

## 적용 시기

### ✅ 적합한 상황

- 소규모 팀 (3~5명)
- 예산이 제한적인 프로젝트
- 빠르게 프로토타입을 만들어야 할 때
- 트래픽이 예측 가능하고 적을 때
- 도메인이 단순할 때

### ❌ 부적합한 상황

- 트래픽이 폭발적으로 증가하는 서비스
- 여러 팀이 독립적으로 개발해야 할 때
- 고가용성이 필요한 시스템
- 빠른 배포 주기가 필요할 때

## 진화 경로

```
초기: 레이어드 아키텍처 (단순)
  ↓
성장: 레이어가 복잡해지고 두꺼워짐
  ↓
문제 발생: 배포가 느려지고, 확장이 어려워짐
  ↓
선택: ① 계속 레이어드 유지 (규모가 크지 않다면)
     ② 서비스 기반 또는 마이크로서비스로 전환
```
