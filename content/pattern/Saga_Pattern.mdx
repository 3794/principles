# Saga Pattern

Saga 패턴은 분산 시스템에서 **여러 서비스에 걸친 트랜잭션을 관리**하는 패턴이다. 마이크로서비스 환경에서 각 서비스는 독립적인 데이터베이스를 가지므로, 전통적인 ACID 트랜잭션을 사용할 수 없다. Saga 패턴은 이 문제를 해결한다.

## 핵심 개념

한 번의 비즈니스 트랜잭션을 여러 단계(saga)로 나누고, 각 단계가 성공하면 다음 단계를 실행한다. 만약 어딘가 실패하면 이전 단계들을 역순으로 보상한다.

```
주문 처리 예시:

① 결제 서비스
   └─ 결제 처리 성공 → ②로 진행

② 재고 서비스
   └─ 재고 차감 실패 → ① 보상 (환불)

③ 배송 서비스
   └─ (실행 안 함)
```

## 두 가지 구현 방식

### 1. Orchestration (중앙 집중식)

한 서비스(Orchestrator)가 다른 서비스들을 제어한다.

```
┌──────────────────┐
│  Order Service   │ ← Orchestrator
│  (지휘자)        │
└────────┬─────────┘
         │
    ┌────┴────┬────────────┐
    ▼         ▼            ▼
Payment   Inventory    Shipping
Service   Service      Service
```

**흐름:**

```
Order Service: "Payment Service, 결제 처리해줘"
Payment Service: "완료" → 상태 저장
Order Service: "Inventory Service, 재고 차감해줘"
Inventory Service: "실패" → Order Service가 Payment Service에 환불 요청
```

**특징:**

- 중앙에서 흐름 제어
- 장점: 흐름이 명확, 디버깅 쉬움
- 단점: Orchestrator가 병목, 강한 결합

### 2. Choreography (이벤트 기반)

각 서비스가 독립적으로 이벤트를 듣고 반응한다.

```
Payment Service
    │ (결제 완료 이벤트 발행)
    ▼
Inventory Service
    │ (재고 차감 완료 이벤트 발행)
    ▼
Shipping Service
    │ (배송 준비 완료 이벤트 발행)
    ▼
Order Service
    (최종 상태 업데이트)
```

**흐름:**

```
Order Service: "결제 요청 이벤트 발행"
Payment Service: (이벤트 감지) → 결제 처리 → "결제 완료 이벤트 발행"
Inventory Service: (이벤트 감지) → 재고 차감 → "재고 차감 완료 이벤트 발행"
Shipping Service: (이벤트 감지) → 배송 준비 → "배송 준비 완료 이벤트 발행"
```

**특징:**

- 서비스들이 이벤트로 소통
- 장점: 느슨한 결합, 확장성 좋음
- 단점: 흐름 추적 어려움, 순환 의존성 위험

## 비교

| 구분      | Orchestration | Choreography   |
| --------- | ------------- | -------------- |
| 흐름 제어 | 중앙 서비스   | 각 서비스 자율 |
| 결합도    | 높음          | 낮음           |
| 흐름 이해 | 쉬움          | 어려움         |
| 테스트    | 비교적 쉬움   | 복잡           |
| 확장성    | 낮음          | 높음           |
| 성능      | 빠름          | 느릴 수 있음   |
| 오류 처리 | 명확함        | 복잡함         |

## 보상 트랜잭션 (Compensating Transaction)

Saga의 핵심은 실패 시 이전 단계를 되돌리는 것이다. 이를 **보상 트랜잭션**이라 한다.

```
정상 흐름:
① 결제 (100만원) → ② 재고 차감 → ③ 배송 준비

실패 흐름 (②에서 실패):
① 결제 (100만원) → ② 재고 차감 실패
                    ↓
              ① 결제 취소 (환불)
```

**주의:** 보상 트랜잭션도 실패할 수 있다. 예를 들어 환불 처리도 실패할 수 있으므로, 재시도 로직과 모니터링이 필수다.

## 적용 시기

### ✅ 적합한 상황

- 마이크로서비스 아키텍처
- 여러 서비스 간 장기 실행 트랜잭션
- 이벤트 기반 시스템
- 높은 가용성이 필요한 시스템

### ❌ 부적합한 상황

- 단일 데이터베이스 시스템 (ACID 트랜잭션 사용)
- 매우 강한 일관성이 필요한 시스템
- 트랜잭션 시간이 매우 짧은 경우
- 보상 로직을 정의하기 어려운 경우

## 주요 고려사항

### 부분 성공 (Partial Success)

일부 서비스는 성공하고 일부는 실패했을 때, 시스템의 상태가 일관되지 않을 수 있다. 이를 **Eventual Consistency(최종 일관성)** 이라 한다.

### 타임아웃

서비스가 응답하지 않을 때 얼마나 기다릴 것인가? 너무 길면 전체 흐름이 느려지고, 너무 짧으면 실패로 간주될 수 있다.

### 모니터링과 로깅

여러 서비스에 걸친 흐름이므로 추적이 어렵다. 각 단계마다 로깅하고 전체 흐름을 추적할 수 있는 구조가 필요하다.

## 실제 사례

### 전자상거래 주문 처리

```
1. Order Service: 주문 생성
2. Payment Service: 결제 처리
3. Inventory Service: 재고 차감
4. Shipping Service: 배송 준비
5. Notification Service: 주문 확인 이메일 발송
```

### 예약 시스템

```
1. Booking Service: 예약 생성
2. Payment Service: 결제
3. Inventory Service: 좌석 확보
4. Notification Service: 예약 확인 알림
```

### 여행 예약

```
1. Trip Service: 여행 계획 생성
2. Flight Service: 항공권 예약
3. Hotel Service: 호텔 예약
4. Car Rental Service: 렌터카 예약
5. Insurance Service: 여행보험 가입
```

## 현대 구현

마이크로서비스와 이벤트 스트리밍 플랫폼에서 Saga 패턴을 많이 본다:

- **Apache Kafka** — 이벤트 스트림 저장소
- **Spring Cloud Stream** — Spring에서 메시지 기반 마이크로서비스
- **AWS Step Functions** — 분산 워크플로우 조율
- **Temporal** — Saga 패턴 구현을 위한 전용 프레임워크
- **Axon Framework** — Java 기반 CQRS/Event Sourcing 프레임워크

특히 대규모 분산 시스템에서는 Saga 패턴이 표준이 되어가고 있다.
