# Happy Path

오류나 예외 없이 모든 입력이 유효하고 모든 단계가 성공하는 **정상적인 흐름 시나리오**다.

## 핵심 개념

시스템이 의도된 대로 동작할 때의 최적 경로를 의미한다. 테스트, 설계, 문서화에서 기준점으로 사용되며, 이후 엣지 케이스와 에러 케이스를 별도로 다룬다.

```
로그인 Happy Path:

① 사용자가 올바른 이메일/비밀번호 입력
   └─ 성공 → ②로 진행

② 서버가 인증 토큰 발급
   └─ 성공 → ③으로 진행

③ 대시보드 페이지로 이동
   └─ 완료
```

## 활용

**테스트**: Happy Path 테스트를 먼저 작성해 기본 기능이 동작함을 확인하고, 이후 Sad Path(실패 경로)를 추가한다.

**설계**: API나 UI 설계 시 Happy Path를 먼저 정의하면 핵심 흐름이 명확해진다.

**문서화**: 사용자 가이드나 API 문서에서 Happy Path 예제를 먼저 보여주는 것이 관례다.

## Happy Path vs Sad Path

| | Happy Path | Sad Path |
|---|---|---|
| 입력 | 유효한 값 | 잘못된 값, 누락, 형식 오류 |
| 상태 | 정상 | 네트워크 오류, DB 장애 등 |
| 결과 | 기대한 성공 응답 | 에러 처리, 롤백 |
