# Space-Based Architecture

Space-Based Architecture(SBA)는 대량의 동시 요청과 트래픽 급증을 처리하기 위해 애플리케이션 상태와 처리를 여러 노드에 분산하는 아키텍처 스타일이다.

이름은 생소하지만 목적은 분명하다. Space-Based Architecture는 일반적인 웹 애플리케이션보다 훨씬 극단적인 트래픽 변동을 처리하기 위해 설계된 특수한 아키텍처다.

예를 들어 콘서트 티켓 예매 시스템을 생각해볼 수 있다.

```
평상시:              동시 사용자 1,000명
콘서트 티켓 오픈 순간:  동시 사용자 1,000,000명
```

이런 상황에서는 애플리케이션 서버를 몇 대 더 늘리는 것만으로는 부족하다. 요청이 결국 중앙 데이터베이스로 몰리기 때문에 DB가 먼저 병목이 되거나 장애가 난다.

전통적인 아키텍처에서는 데이터베이스가 병목이 되기 쉽다. Space-Based Architecture는 요청 처리에 필요한 데이터를 메모리 기반의 **shared space**에 분산해두고, 여러 처리 노드가 그 공간을 함께 사용하도록 만든다.

핵심 아이디어는 "중앙 데이터베이스에 모든 요청이 몰리지 않게 하고, 처리와 상태를 확장 가능한 공간에 분산한다"는 것이다.

## Structure(구조)

```
┌──────────────────────────────────────────────┐
│              Client / API Gateway            │
└──────────────────────┬───────────────────────┘
                       │
        ┌──────────────┼──────────────┐
        │              │              │
┌───────▼───────┐ ┌────▼────────┐ ┌───▼─────────┐
│ Processing    │ │ Processing  │ │ Processing  │
│ Unit A        │ │ Unit B      │ │ Unit C      │
└───────┬───────┘ └────┬────────┘ └───┬─────────┘
        │              │              │
┌───────▼──────────────▼──────────────▼─────────┐
│              Virtualized Middleware            │
│  ┌────────────┐ ┌────────────┐ ┌─────────────┐ │
│  │ Data Grid  │ │ Messaging  │ │ Processing  │ │
│  │            │ │ Grid       │ │ Grid        │ │
│  └────────────┘ └────────────┘ └─────────────┘ │
└──────────────────────┬────────────────────────┘
                       │
              ┌────────▼────────┐
              │ Database /      │
              │ External System │
              └─────────────────┘
```

## Core Concepts(핵심 개념)

- **Processing Unit** - 애플리케이션 로직을 실행하는 독립 배포 단위
- **Virtualized Middleware** - 데이터, 메시징, 처리, 배포를 분산 환경에서 관리하는 미들웨어 계층
- **Data Grid** - 여러 노드에 분산된 인메모리 데이터 저장 공간
- **Messaging Grid** - 처리 노드 사이의 비동기 메시지 전달 계층
- **Processing Grid** - 작업을 여러 노드에 분산 실행하는 계층
- **Deployment Manager** - 처리 노드의 배포, 복제, 확장을 관리하는 구성 요소

Space-Based Architecture에서 "space"는 여러 처리 노드가 공유하는 분산 메모리 공간을 의미한다. 단순한 캐시라기보다 애플리케이션의 활성 상태와 작업 흐름을 담는 실행 공간에 가깝다.

## 동작 방식

```
1. 요청이 API Gateway로 들어온다
2. 요청이 여러 Processing Unit 중 하나로 라우팅된다
3. Processing Unit은 필요한 데이터를 Data Grid에서 읽는다
4. 처리 결과를 다시 Data Grid에 반영한다
5. Data Writer가 변경 내용을 비동기적으로 DB에 저장한다
6. 다른 Processing Unit은 같은 Space를 통해 최신 상태를 참조한다
```

이 구조에서는 모든 요청이 데이터베이스로 직접 가지 않는다. 많은 읽기와 쓰기가 메모리 기반 공간에서 처리되고, 영속 저장소와의 동기화는 비동기로 분리될 수 있다.

## Simple Example(간단한 예제)

아래 예제는 주문 요청을 처리할 때 데이터베이스에 바로 쓰지 않고, 인메모리 공간에 먼저 반영한 뒤 비동기 writer가 영속화하는 단순화된 구조이다.

```typescript
// space/OrderSpace.ts
export interface Order {
  id: string;
  userId: string;
  amount: number;
  status: "pending" | "confirmed";
}

export class OrderSpace {
  private orders = new Map<string, Order>();
  private dirtyOrderIds = new Set<string>();

  write(order: Order) {
    this.orders.set(order.id, order);
    this.dirtyOrderIds.add(order.id);
  }

  read(orderId: string) {
    return this.orders.get(orderId);
  }

  pullDirtyOrders() {
    const orders = Array.from(this.dirtyOrderIds)
      .map((id) => this.orders.get(id))
      .filter((order): order is Order => Boolean(order));

    this.dirtyOrderIds.clear();
    return orders;
  }
}
```

```typescript
// app/OrderProcessingUnit.ts
import { OrderSpace } from "../space/OrderSpace";

export class OrderProcessingUnit {
  constructor(private orderSpace: OrderSpace) {}

  async createOrder(userId: string, amount: number) {
    const order = {
      id: crypto.randomUUID(),
      userId,
      amount,
      status: "pending" as const,
    };

    this.orderSpace.write(order);

    return order;
  }

  async confirmOrder(orderId: string) {
    const order = this.orderSpace.read(orderId);

    if (!order) {
      throw new Error(`Order not found: ${orderId}`);
    }

    const confirmedOrder = { ...order, status: "confirmed" as const };
    this.orderSpace.write(confirmedOrder);

    return confirmedOrder;
  }
}
```

```typescript
// persistence/OrderWriter.ts
import { OrderSpace } from "../space/OrderSpace";

export class OrderWriter {
  constructor(private orderSpace: OrderSpace) {}

  async flush() {
    const dirtyOrders = this.orderSpace.pullDirtyOrders();

    await Promise.all(
      dirtyOrders.map(async (order) => {
        console.log(`Persist order ${order.id} to database`);
        // 실제 시스템에서는 DB upsert를 수행한다.
      }),
    );
  }
}
```

실제 Space-Based Architecture에서는 `OrderSpace` 같은 단순 객체 대신 Hazelcast, Apache Ignite, GigaSpaces 같은 인메모리 데이터 그리드나 분산 캐시 기술을 사용할 수 있다.

## 구성 요소

### Processing Unit

비즈니스 로직을 실행하는 배포 단위이다. 각 Processing Unit은 동일한 코드를 가진 복제본일 수 있고, 트래픽 증가에 따라 수평 확장된다.

```
Processing Unit = Business Logic + Local State Access + Messaging Client
```

### Data Grid

활성 데이터를 메모리에 분산 저장한다. 데이터가 여러 노드에 복제되거나 파티셔닝되어 특정 노드 장애에도 계속 처리할 수 있다.

### Messaging Grid

Processing Unit 간 이벤트나 작업 메시지를 전달한다. 직접 호출을 줄이고 작업을 비동기화하는 데 사용한다.

### Data Writer

Space에 반영된 변경을 데이터베이스에 저장한다. 보통 비동기로 동작하므로 쓰기 지연과 최종 일관성을 고려해야 한다.

### Data Reader

데이터베이스나 외부 시스템에서 필요한 데이터를 Space로 적재한다. 자주 쓰는 데이터를 미리 올려두거나, 캐시 미스가 발생했을 때 데이터를 가져온다.

## 장점

- 데이터베이스 병목을 줄일 수 있음
- 트래픽 급증에 수평 확장으로 대응하기 쉬움
- 메모리 기반 처리로 낮은 지연 시간을 기대할 수 있음
- 처리 노드를 추가하거나 제거하면서 탄력적으로 운영할 수 있음
- 일부 노드 장애가 전체 장애로 이어지는 것을 줄일 수 있음
- 읽기와 쓰기를 분산해 고부하 워크로드에 유리함

## 단점

- 데이터 동기화와 일관성 관리가 어려움
- 분산 캐시, 데이터 그리드, 메시징 인프라 운영 비용이 큼
- 장애 상황에서 데이터 유실, 중복 처리, 순서 문제가 생길 수 있음
- 디버깅과 테스트가 단일 데이터베이스 중심 구조보다 복잡함
- 모든 데이터를 메모리에 둘 수 없으므로 데이터 분할 전략이 필요함
- 강한 트랜잭션 일관성이 중요한 도메인에는 맞지 않을 수 있음

## Design Guidelines(설계 지침)

### 1. 데이터베이스 병목이 실제 문제인지 확인한다

Space-Based Architecture는 복잡하다. 단순히 "확장성이 좋아 보인다"는 이유로 선택하면 운영 비용만 늘어난다. 먼저 병목이 데이터베이스인지, 애플리케이션 로직인지, 외부 API인지 확인해야 한다.

### 2. Space에 둘 데이터를 제한한다

모든 데이터를 메모리에 올릴 수는 없다. 빠른 접근이 필요한 활성 데이터, 세션 상태, 진행 중인 주문, 최근 조회 데이터처럼 고부하 경로에 있는 데이터부터 분리해야 한다.

```
적합한 데이터:
현재 진행 중인 주문
실시간 재고 카운터
최근 검색 결과
사용자 세션

부적합한 데이터:
장기 보관 로그
대용량 분석 원천 데이터
거의 접근하지 않는 과거 데이터
```

### 3. 최종 일관성을 명시적으로 설계한다

Space의 상태와 데이터베이스의 상태가 항상 즉시 같지는 않을 수 있다. 사용자에게 보여줄 상태, 재시도 정책, 보상 처리, 동기화 실패 알림을 설계해야 한다.

### 4. 쓰기 경로를 멱등하게 만든다

Data Writer가 재시도되면 같은 변경이 여러 번 저장될 수 있다. DB upsert, 이벤트 id, 버전 번호를 사용해 중복 처리를 방지해야 한다.

```typescript
interface PersistedChange {
  changeId: string;
  entityId: string;
  version: number;
  payload: unknown;
}
```

### 5. 관찰 가능성을 먼저 준비한다

분산된 Processing Unit과 Data Grid에서 문제가 생기면 원인 추적이 어렵다. 요청 id, 상태 변경 로그, grid metric, replication lag, writer 실패율을 관찰할 수 있어야 한다.

## 적용 시기

### 적합한 상황

- 짧은 시간에 트래픽이 폭발적으로 증가하는 서비스
- 데이터베이스가 명확한 병목인 고부하 시스템
- 낮은 지연 시간이 중요한 실시간 서비스
- 사용자 세션, 주문, 예약, 게임 상태처럼 활성 상태가 많은 시스템
- 읽기와 쓰기 요청이 많고 수평 확장이 중요한 시스템

### 부적합한 상황

- 단순한 CRUD 애플리케이션
- 강한 ACID 트랜잭션이 핵심인 금융 원장 시스템
- 운영할 분산 인프라 경험이 부족한 팀
- 데이터 양이 크지만 대부분 콜드 데이터인 시스템
- 트래픽이 예측 가능하고 데이터베이스 튜닝만으로 충분한 시스템
