# Event-driven Architecture

Event-driven Architecture(EDA)는 시스템의 흐름을 직접 호출이 아니라 **이벤트의 발생과 반응**으로 구성하는 아키텍처 스타일이다. 어떤 서비스가 "무슨 일이 일어났다"는 이벤트를 발행하면, 그 이벤트에 관심 있는 다른 서비스나 컴포넌트가 비동기적으로 반응한다.

핵심은 요청자가 처리자를 직접 알지 않아도 된다는 것이다. 주문 서비스는 "주문이 생성되었다"는 사실만 발행하고, 이메일 발송, 재고 차감, 분석 수집, 포인트 적립은 각자의 consumer가 처리한다.

## Structure(구조)

```
┌──────────────┐      ┌──────────────────┐      ┌────────────────┐
│ Event Source │─────▶│ Event Router     │─────▶│ Event Consumer │
│ Order        │      │ Broker / Stream  │      │ Email          │
└──────────────┘      └──────────────────┘      └────────────────┘
                              │
                              │                ┌────────────────┐
                              ├───────────────▶│ Event Consumer │
                              │                │ Inventory      │
                              │                └────────────────┘
                              │
                              │                ┌────────────────┐
                              └───────────────▶│ Event Consumer │
                                               │ Analytics      │
                                               └────────────────┘
```

## Core Concepts(핵심 개념)

- **Event** - 이미 발생한 중요한 사실. 예: `OrderCreated`, `PaymentFailed`
- **Producer** - 이벤트를 발행하는 주체
- **Consumer** - 이벤트를 구독하고 처리하는 주체
- **Event Router** - 이벤트를 전달하는 중간 계층. 메시지 브로커, 이벤트 버스, 스트리밍 플랫폼 등이 여기에 해당한다
- **Event Channel** - 이벤트가 흐르는 경로. 토픽, 큐, 스트림으로 표현된다
- **Event Handler** - 특정 이벤트를 받아 실제 작업을 수행하는 코드

이벤트 이름은 명령형보다 과거형 사실로 표현하는 것이 좋다.

```
추천 예제:
OrderCreated
PaymentCaptured
UserEmailChanged
```

## 동작 방식

```
1. 사용자가 주문을 생성한다
2. Order Service가 주문을 저장한다
3. Order Service가 OrderCreated 이벤트를 발행한다
4. Inventory Service가 재고를 차감한다
5. Email Service가 주문 확인 메일을 보낸다
6. Analytics Service가 주문 지표를 기록한다
```

각 consumer는 서로를 알지 못한다. 주문 서비스도 이메일 서비스나 분석 서비스의 존재를 몰라도 된다.

## Simple Example(간단한 예제)

주문 생성 후 여러 후속 작업을 이벤트로 분리한 예제이다.

```typescript
// events/DomainEvent.ts
export interface DomainEvent<T = unknown> {
  id: string;
  type: string;
  occurredAt: Date;
  payload: T;
}

export interface OrderCreatedPayload {
  orderId: string;
  userId: string;
  totalAmount: number;
}
```

```typescript
// events/EventBus.ts
import { DomainEvent } from "./DomainEvent";

type EventHandler<T = unknown> = (event: DomainEvent<T>) => Promise<void>;

export class EventBus {
  private handlers = new Map<string, EventHandler[]>();

  subscribe(eventType: string, handler: EventHandler) {
    const handlers = this.handlers.get(eventType) ?? [];
    handlers.push(handler);
    this.handlers.set(eventType, handlers);
  }

  async publish<T>(event: DomainEvent<T>) {
    const handlers = this.handlers.get(event.type) ?? [];

    await Promise.all(
      handlers.map((handler) =>
        handler(event).catch((error) => {
          console.error(`Failed to handle ${event.type}`, error);
        }),
      ),
    );
  }
}
```

```typescript
// orders/OrderService.ts
import { EventBus } from "../events/EventBus";
import { DomainEvent, OrderCreatedPayload } from "../events/DomainEvent";

export class OrderService {
  constructor(private eventBus: EventBus) {}

  async createOrder(userId: string, totalAmount: number) {
    const orderId = crypto.randomUUID();

    // 실제 시스템에서는 여기서 주문을 DB에 저장한다.
    const event: DomainEvent<OrderCreatedPayload> = {
      id: crypto.randomUUID(),
      type: "OrderCreated",
      occurredAt: new Date(),
      payload: { orderId, userId, totalAmount },
    };

    await this.eventBus.publish(event);

    return { orderId };
  }
}
```

```typescript
// handlers/OrderCreatedHandlers.ts
import { DomainEvent, OrderCreatedPayload } from "../events/DomainEvent";

export async function reserveInventory(
  event: DomainEvent<OrderCreatedPayload>,
) {
  console.log(`Reserve inventory for order ${event.payload.orderId}`);
}

export async function sendOrderEmail(event: DomainEvent<OrderCreatedPayload>) {
  console.log(`Send order email to user ${event.payload.userId}`);
}

export async function recordAnalytics(event: DomainEvent<OrderCreatedPayload>) {
  console.log(`Track order amount ${event.payload.totalAmount}`);
}
```

```typescript
// main.ts
import { EventBus } from "./events/EventBus";
import { OrderService } from "./orders/OrderService";
import {
  recordAnalytics,
  reserveInventory,
  sendOrderEmail,
} from "./handlers/OrderCreatedHandlers";

const eventBus = new EventBus();

eventBus.subscribe("OrderCreated", reserveInventory);
eventBus.subscribe("OrderCreated", sendOrderEmail);
eventBus.subscribe("OrderCreated", recordAnalytics);

const orderService = new OrderService(eventBus);
await orderService.createOrder("USER-001", 50000);
```

이 예제에서는 인메모리 `EventBus`를 사용했지만, 실제 분산 시스템에서는 Kafka, RabbitMQ, SNS/SQS, NATS 같은 메시징 인프라를 사용하는 경우가 많다.

## Topologies(토폴로지)

### Mediator Topology

중앙 중재자가 전체 이벤트 흐름을 조정하는 방식이다.

```
Producer → Event Mediator → Step 1 → Step 2 → Step 3
```

복잡한 프로세스의 순서와 보상 작업을 명시적으로 관리하기 좋다. 대신 중재자가 비대해지면 병목과 결합 지점이 된다.

### Broker Topology

이벤트 브로커가 이벤트 전달만 담당하고, consumer들이 독립적으로 반응하는 방식이다.

```
Producer → Event Broker → Consumer A
                        → Consumer B
                        → Consumer C
```

느슨한 결합과 확장성에 유리하다. 대신 전체 흐름을 한눈에 파악하기 어렵고, 이벤트 추적과 관찰 가능성이 중요해진다.

## 장점

- 서비스 간 결합도를 낮출 수 있음
- 새로운 consumer를 추가해도 생산자를 수정하지 않아도 됨
- 비동기 처리로 응답 시간을 줄일 수 있음
- 트래픽이 많은 작업을 큐나 스트림으로 흡수할 수 있음
- 장애가 한 컴포넌트에서 전체 시스템으로 바로 전파되는 것을 줄일 수 있음
- 이벤트 로그를 분석, 감사, 재처리에 활용할 수 있음

## 단점

- 처리 흐름이 분산되어 디버깅이 어려움
- 이벤트 순서, 중복 처리, 유실 방지 같은 문제가 생김
- 최종 일관성(Eventual Consistency)을 받아들여야 할 수 있음
- 메시지 브로커나 스트리밍 플랫폼 운영 비용이 추가됨
- 이벤트 스키마가 바뀔 때 하위 호환성을 관리해야 함
- 장애 원인을 추적하려면 로그, 트레이싱, 메트릭이 잘 갖춰져야 함

## Design Guidelines(설계 지침)

### 1. 이벤트는 사실을 표현한다

이벤트는 처리하라는 명령이 아니라 이미 일어난 사실이어야 한다.

```
Command: ShipOrder
Event: OrderShipped
```

명령은 특정 처리자를 가정하지만, 이벤트는 여러 consumer가 각자의 방식으로 반응할 수 있다.

### 2. 이벤트는 필요한 데이터를 충분히 담는다

consumer가 이벤트를 처리하기 위해 생산자에게 다시 조회해야 한다면 결합도가 다시 높아질 수 있다. 이벤트에는 consumer가 독립적으로 판단할 수 있는 핵심 데이터를 담는 편이 좋다.

다만 모든 데이터를 이벤트에 넣으면 스키마 변경 비용이 커진다. 이벤트는 "작지만 의미 있게" 설계해야 한다.

### 3. consumer는 멱등하게 만든다

분산 메시징에서는 같은 이벤트가 두 번 전달될 수 있다. consumer는 같은 이벤트를 여러 번 처리해도 결과가 깨지지 않도록 만들어야 한다.

```typescript
async function handlePaymentCaptured(event: PaymentCaptured) {
  if (await alreadyProcessed(event.id)) {
    return;
  }

  await markAsProcessed(event.id);
  await completeOrder(event.payload.orderId);
}
```

### 4. 이벤트 스키마 버전을 관리한다

이벤트는 한번 발행되면 여러 consumer가 사용한다. 필드를 제거하거나 의미를 바꾸면 consumer가 깨질 수 있다.

- 새 필드는 optional로 추가한다
- 기존 필드의 의미를 바꾸지 않는다
- 큰 변경은 새 이벤트 타입이나 새 버전으로 분리한다
- 오래된 consumer가 새 이벤트를 받아도 실패하지 않게 한다

### 5. 관찰 가능성을 먼저 설계한다

Event-driven 시스템에서는 한 요청이 여러 이벤트와 consumer로 흩어진다. `correlationId`, 구조화 로그, 분산 트레이싱, dead letter queue를 준비해야 운영할 수 있다.

## 적용 시기

### 적합한 상황

- 여러 서비스가 같은 사건에 독립적으로 반응해야 할 때
- 주문, 결제, 배송, 알림처럼 후속 처리가 많은 도메인
- 순간 트래픽을 큐나 스트림으로 흡수해야 할 때
- 실시간 분석, 감사 로그, 알림, 데이터 동기화가 중요한 시스템
- 서비스 간 직접 호출이 많아 결합도가 높아진 시스템

### 부적합한 상황

- 단순한 CRUD 애플리케이션
- 강한 즉시 일관성이 핵심인 트랜잭션
- 이벤트 흐름을 운영하고 추적할 역량이 부족한 팀
- 실패 재처리, 중복 처리, 순서 보장을 설계할 시간이 없는 프로젝트
